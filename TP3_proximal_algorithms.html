Structure of an optimization program¶

An optimization program can be practically divided into three parts:

  • the run environment, in which you test, run your program, and display results.
  • the problem part, which contains the function oracles, problem constraints, etc.
  • the algorithmic part, where the algorithms are coded.

The main interest of such division is that these parts are interchangeable, meaning that, for instance, the algorithms of the third part can be used of a variety of problems. That is why such a decomposition is widely used.

In the present lab, you will use this division:

  • TP3_Proximal_algorithms.ipynb will be the run environment
  • logistic_regression.ipynb will be the considered logistic regression problem for this lab
  • algoProx.ipynb will contain the proximal algorithms studied in this lab

Tout d'abord pour $ln(\lambda) = -1, 0, 1$ nous avons trois mauvais scores identiques, que ce soit en prédiction sur les données d'entraînement ou les les données de tests. Ceci s'explique par le fait que pour ces valeurs de $ln(\lambda)$ le support du vecteurs $x$, sélectionné par l'algorithme, est vide. Toutes les coordonnées du vecteurs sont nulles, ainsi $\forall (a_i,b_i)\in \mathbb{R}^d \times \mathcal{O}$ le produit scalaire $ \langle a_i , x \rangle $ donne 0 ce qui donne toujours un $p_1(a) = \frac{1}{2}$. Dans les fonctions de prédictions, nous tombons la deuxième branche conditionnelle, ce qui fait que ne prédit que $b_i$ négatif. Ainsi ces valeurs ne représentent que le nombre d'observations de label $-1$.

Pour ce qui est du reste du graphe, la précision de test est plus faible que la précision d'entraînement. Ce qui est tout à fait cohérent car le modèle, x, est une solution approchée d'une solution d'un problème d'optimisation sur les données d'entraînement. C'est donc un modèle qui donnera de meilleurs résultats sur des jeux de données qui sont très proches du jeu de données d'entraînement.

La précision d'entraînement reste au-dessus de $90\%$ et est décroissante en $ln \lambda$. Plus lambda est petit, plus la précision est bonne.

La précision de tester reste varie autour des $85\%$. Elle atteint un son maximum empirique pour $\lambda = 2^{-6}$ avec $90\%$ de bonne réponse, avant de redescendre à un taux de bonne réponse légèrement supérieur à $85\%$ pour \lambda petit$.

Sur $[\lambda = 2^{-12},\lambda = 2^{-6}]$ la précision du modèle sur le jeu de données d'entraînement augmente quand celle sur le jeu de données de test diminue. On peut penser qu'en dessous $\lambda = 2^{-6}$ il y a un sur-apprentissage du modèle qui implique une moins bonne robustesse de ce dernier quand on passe à de nouvelles données. Ce sur-apprentissage apparaît car $\lambda_1$ est trop petit, son poids dans la fonction objective devient négligeable devant la fonction de perte logistique. Or, c'est cette dernière qui quantifie la précision de notre modèle sur les données d'entrainements. Donc, notre modèle optimisé surtout sa précision sur les données d'entraînement.

Quand $\lambda_1$ augmente, son poids dans la fonction objective augmente et de ce fait on donne moins d'importance à la fonction de perte logistique. Or, c'est cette dernière qui quantifie la précision de notre modèle sur les données d'entrainements. D'où, la perte de précision quand $\lambda_1$ augmente.

L est égale à  32.965468577853414 .

$||x_{k+1}-x^*||\leq \left(\dfrac{1-\kappa}{1+\kappa}\right)^k||x_{0}-x^*||$ Avec $\kappa$ = $\dfrac{\lambda_2}{L}\leq 1$ et $x^*$ la solution du problème.

D'après cette inégalité, il est cohérent que la vitesse de convergence augmente (le nombre d'itérations diminue) quand $\lambda_2$ augmente.

Comme $\kappa \geq 0 $, on a $||x_{k+1}-x^*||\leq \left(1-\kappa\right)^k||x_{0}-x^*||$

Si l'on a un critère d'arrêt du type $||x_{k+1}-x^*||\leq \epsilon $ avec $ \epsilon \gt 0$, l'algorithme s'arrête quand : $ \left(1-\kappa\right)^k||x_{0}-x^*|| \leq \epsilon$.

On passe au log dans l'inégalité, on obtient: $ k \log{\left(1-\kappa\right)} + \log{||x_{0}-x^*||} \leq \log{\epsilon}$

Soit : $ k \log{\left(1-\kappa\right)} \leq \log{\frac{\epsilon}{||x_{0}-x^*||}} = -C$, une constante positive) ($\epsilon \leq ||x_{0}-x^*||$ sinon l'algorithme s'arrête avec 0 itération. )

Comme on fait varier $\lambda_1$ dans $]0,2[$ et que $L \approx 33 $ on peut considérer que $\kappa$ est très petit devant $1$.

Donc : $$ k \leq \frac{-C}{\log{\left(1-\kappa\right)}}$$

or $\log(1-\kappa) \sim -\kappa$ quand $\kappa$ tend vers 0.

Donc $$ k \leq \frac{C}{\kappa + O(\kappa)}$$

Ainsi k est majoré par une fonction équivalente à $$\frac{C}{\kappa} = \frac{C\times L}{\lambda_2}$$

Donc il n'est pas surprenant de retrouver une décroissance en $\frac{1}{\lambda_2}$

Un autre critère d'arrêt ?¶

Un autre critère d'arrêt peut se baser sur le score de prédiction de notre modèle sur les données d'entraînements et/ou de tests. Il est clair que ne prendre en compte que le score de prédiction de notre modèle sur les données d'entrainements va aboutir à un sur-apprentissage de notre modèle. Afin d'éviter cela, nous allons utiliser une méthode de cross-validation. Le modèle va être entraîné sur les données d'entraînement puis être validé par nos données de tests (qui vont donc devenir nos données de validation). A chaque "pas d'itérations", par exemple tous les 10, 20, 50, 100 ... itérations, le modèle sera évalué sur le jeu de données de validation. Une fois que la précision sur les données aura atteint un palier, l'algorithme s'arrêtera. Il faudra ensuite un troisième jeu de données afin de tester notre modèle (car il est normal qu'il soit plus performant sur les données de validations à cause de ce procédé).
