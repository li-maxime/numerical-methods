Structure of an optimization program¶

An optimization program can be practically divided into three parts:

  • the run environment, in which you test, run your program, and display results.
  • the problem part, which contains the function oracles, problem constraints, etc.
  • the algorithmic part, where the algorithms are coded.

The main interest of such division is that these parts are interchangeable, meaning that, for instance, the algorithms of the third part can be used of a variety of problems. That is why such a decomposition is widely used.

In the present lab, you will use this division:

  • TP3_Proximal_algorithms.ipynb will be the run environment
  • logistic_regression.ipynb will be the considered logistic regression problem for this lab
  • algoProx.ipynb will contain the proximal algorithms studied in this lab

$||x_{k+1}-x^*||\leq \left(\dfrac{1-\kappa}{1+\kappa}\right)^k||x_{0}-x^*||$ Avec $\kappa$ = $\dfrac{\lambda_2}{L}\leq 1$ et $x^*$ la solution du problme.

D'aprs cette ingalit, il est cohrent que la vitesse de convergence augmente (le nombre d'itrations diminue) quand $\lambda_2$ augmente.

Comme $\kappa \geq 0 $, on a $||x_{k+1}-x^*||\leq \left(1-\kappa\right)^k||x_{0}-x^*||$

Si l'on a un critre d'arrt du type $||x_{k+1}-x^*||\leq \epsilon $ avec $ \epsilon \gt 0$, l'algorithme s'arrte quand : $ \left(1-\kappa\right)^k||x_{0}-x^*|| \leq \epsilon$.

On passe au log dans l'ingalit, on obtient: $ k \log{\left(1-\kappa\right)} + \log{||x_{0}-x^*||} \leq \log{\epsilon}$

Soit : $ k \log{\left(1-\kappa\right)} \leq \log{\frac{\epsilon}{||x_{0}-x^*||}} = -C$, une constante positive) ($\epsilon \leq ||x_{0}-x^*||$ sinon l'algorithme s'arrte avec 0 itration. )

Comme on fait varier $\lambda_1$ dans $]0,2[$ et que $L \approx 33 $ on peut considrer que $\kappa$ est trs petit devant $1$.

Donc : $$ k \leq \frac{-C}{\log{\left(1-\kappa\right)}}$$

or $\log(1-\kappa) \sim -\kappa$ quand $\kappa$ tend vers 0.

Donc $$ k \leq \frac{C}{\kappa + O(\kappa)}$$

Ainsi k est major par une fonction quivalente $$\frac{C}{\kappa} = \frac{C\times L}{\lambda_2}$$

Donc il n'est pas surprenant de retrouver une dcroissance en $\frac{1}{\lambda_2}$

Un autre critère d'arrêt ?¶

Un autre critre d'arrt peut se baser sur le score de prdiction de notre modle sur les donnes d'entranements et/ou de tests. Il est clair que ne prendre en compte que le score de prdiction de notre modle sur les donnes d'entrainements va aboutir un sur-apprentissage de notre modle. Afin d'viter cela, nous allons utiliser une mthode de cross-validation. Le modle va tre entran sur les donnes d'entranement puis tre valid par nos donnes de tests (qui vont donc devenir nos donnes de validation). A chaque "pas d'itrations", par exemple tous les 10, 20, 50, 100 ... itrations, le modle sera valu sur le jeu de donnes de validation. Une fois que la prcision sur les donnes aura atteint un palier, l'algorithme s'arrtera. Il faudra ensuite un troisime jeu de donnes afin de tester notre modle (car il est normal qu'il soit plus performant sur les donnes de validations cause de ce procd).
